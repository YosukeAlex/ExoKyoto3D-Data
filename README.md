# ExoKyoto3D-Data

Public data distribution channel for the **ExoKyoto3D / 4D / 4Dalpha** macOS apps.

## Files

| File | Purpose |
|---|---|
| `version.json`                       | Manifest read by every running app at launch |
| `data/latest/ExoKyotoDataF.bin`      | Current exoplanet catalogue (EKDBIN1 binary, ~12 MB, ~6480 rows) |
| `data/past/ExoKyotoDataF<YYYYMMDDNN>.bin` | Archived previous versions |

## How the app discovers updates

1. App launches.
2. Background thread `curl`s `version.json` (this file).
3. Compares the `data_version` field with the stamp baked into the binary at build time.
4. If different → download `binary_url` and apply.

See `UPDATE_PIPELINE.md` in this repo for the full update-publishing workflow.

## Producing a new release

Maintainer-side, in the `gaia/` workspace (not in this repo):

```bash
cd <gaia>
python3 exokyoto_update_pipeline.py \
    --in-xlsx <new>.xlsx \
    --data-version "YYYY.MM.DD<suffix>-<channel>" \
    --release-notes "<changes>"
```

Then `git push` from this directory (which is the local clone of this repo).

## Update flow (summary)

- App launches → background thread `curl`s `version.json` here
- Compares `data_version` against the sidecar `csvin/ExoKyotoDataF.bin.version`
  (or, on first run, against the compile-time stamp)
- If different → dialog asks the user; on confirmation, downloads
  `data/latest/ExoKyotoDataF.bin`, swaps it into the app bundle, writes the
  new sidecar, and offers to auto-restart
- Versioning: `data_version` is auto-generated by the maintainer pipeline as
  `YYYY.MM.DD-HHMM-JST` (Japan-time minute resolution)
